## The identity authority (design v2, primary-56d1.11)

The daemon is the one logical allocation authority per deployment (settled: "seat it centrally in sema"). Its mandate is the keystone's two laws â€” never re-mint an identity for the same declared thing, never rebind an existing identity to a different thing â€” and `Request::BindIdentities` enforces them as durable semantics.

- **Two identity levels, both authority-minted.** A schema-whole handle mints a globally-unique, never-reused universe id (`UniverseRegistry`, the authoritative universe path replacing `structural-codec`'s `FIXTURE_UNIVERSE(0)` placeholder). Within a universe, each declared thing keyed by its `DeclaredKey` mints a never-reused local `TypeIdentity` or binds its existing one (`UniverseState`).
- **Bind-or-mint = the two laws.** `UniverseState::bind_or_mint` is the enforcement point. `MintOrBind` returns the existing identity for a key it already bound (law 1's anti-remint) â€” a changed shape under it is a Â§3 version-advance keeping the id â€” and mints only for a genuinely new key. `Continue(identity)` rebinds an existing identity (a rename/move) only when its recorded shape matches; a different shape is the law-2 rejection `IdentityRebindRejected`.
- **Durable with the same guarantee as documents.** Authority state lives in its own `identity_authority_records` table (the document/allocator schema is untouched), and each bind persists registry + universe state as one atomic `CommitRequest` â€” the same single-commit durability documents get, so a restart re-issues nothing. The rejection path writes nothing, so a rejected bind never disturbs on-disk identities. `tests/identity_authority.rs` proves the keystone across a fresh process, carried-declaration stability, the law-2 rejection, never-reused universes across restart, and the empty-set rejection.

## Revisable leans
- One closed typed record family stores document versions and allocator cursors. Split families only when different retention or schema evolution requires it.
- The Unix socket and `/tmp/new-language-engine` defaults isolate prototype state; production Spirit paths are never used.
- Fixture allocation is monotonic per explicit scope. This deliberately says nothing about permanent split/merge lineage.
- **Fixture-scope allocation is a compatibility lean beside the authoritative mint.** `AllocateIdentifiers` / `StorageRecord::Allocator` (monotonic `IdentifierBlock`s per `FixtureScope`) is retained as a marked compatibility path; the authoritative identity path is now `BindIdentities`, which mints universe and type identities enforcing the two laws. Revise by retiring the fixture-scope allocator once all callers bind through the authority.
- **Authority-layer keys are named leans (design v2).** "The same declared thing" keys on the declared name (`DeclaredKey`), and a schema-whole's identity is an opaque author handle (`SchemaWholeHandle`) that mints a universe rather than carrying an independently minted whole-id â€” the v2 `declared-key-is-name` (L4) and `whole-handle-opaque` (L1) leans, owned by the `signal-sema-storage` contract. The law-2 discriminator is the recorded `DeclaredShape`: a shape change under a `MintOrBind` key is a version-advance (same id), while a `Continue` to a different shape is a rebind (rejected). Revise when an authoring surface carries explicit per-declaration bind/mint markers.
- **The follow-up equivalence slice is deliberately out of scope here.** This daemon mints and binds identities; turning parsed schema text into a bind request and building `core-schema`'s universe from the reply (so two front ends reach the same content hash) is the schema-engine / native-ingestion wiring left for the next slice.
- **Central-Sema-daemon storage axis.** One central `sema-storage` daemon is the sole durable write authority; the schema, nomos, and logos daemons run as stateless socket clients that hold no durable state and persist only through it. The charter's fuller vision instead seats a `sema-engine` inside each component daemon (per-daemon storage). This lean keeps storage central for the prototype. Revise it when a component needs independent durable state, per-daemon retention or schema-evolution divergence, or write throughput a single central daemon cannot serve â€” at which point storage seating moves per-daemon.
- The *allocation-authority* seat is settled, not a lean: the psyche ruled the id-allocation authority central-in-Sema (2026-07-17, "yes, seat it centrally in sema"; bead `primary-56d1.11`). Only the *storage* seating in the entry above remains revisable; keep the two distinct.
